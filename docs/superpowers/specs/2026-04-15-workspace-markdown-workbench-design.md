# Open Cowork 工作区文件面板与 Markdown 工作台设计

## 1. 背景

当前 Open Cowork 的左侧主结构是会话历史，工作目录只在欢迎页和上下文面板中以弱提示存在。这和用户期望的使用路径不一致：用户希望像 VS Code 一样，先进入某个工作区，再在固定的文件树中浏览文件，同时保留中间聊天能力，并在右侧直接编辑和预览 Markdown 文件。

这次设计将应用主界面从“会话历史优先”调整为“工作区优先”，把工作区、聊天、文件工作台三者分成清晰的结构边界。

## 2. 目标

本次设计目标如下：

- 应用启动后优先恢复上次工作区；如果没有可用工作区，则要求用户先选择工作区。
- 主界面改为三栏布局：左侧工作区文件面板，中间聊天窗口，右侧文件主视图。
- 左侧文件面板展示完整目录树，并实时反映当前工作区的文件变化。
- 左侧底部保留历史会话，但降级为可折叠的辅助区域。
- 右侧主视图第一期只支持 `.md` 文件，支持多标签、编辑、预览、自动保存和手动保存。
- 非 `.md` 文件点击后不在主视图中打开，只给出明确提示。
- 提供明显的“打开工作区”入口，方便用户随时切换工作区。

## 3. 非目标

本次不包含以下内容：

- 不支持非 Markdown 文件的应用内编辑或预览。
- 不引入 Monaco、CodeMirror 等重量级编辑器内核，第一期使用轻量编辑实现。
- 不做复杂的文件冲突合并、diff 视图或版本比较。
- 不重写现有会话数据模型，不把“工作区切换”强制同步到所有历史会话。
- 不把 Open Cowork 完整转成 IDE，聊天仍然是中间核心能力之一。

## 4. 用户确认后的产品约束

以下行为已经在需求澄清阶段确认：

- 左侧是文件面板，而不是会话历史主栏。
- 左侧文件面板展示 IDE 风格的完整目录树。
- 中间偏左区域是对话窗口。
- 右侧大区域是主视图，用来编辑和浏览文件。
- 历史会话放在左侧底部，作为可折叠区域。
- 应用记住上次工作区，重启后直接恢复；同时提供“打开工作区”按钮以便切换。
- `.md` 文件在应用内部编辑和预览。
- 非 `.md` 文件不打开，只提示当前暂不支持。
- Markdown 采用多标签模式。
- 保存策略为“自动保存为主，同时保留保存按钮和未保存提示”。

## 5. 总体信息架构

应用主壳调整为三栏结构：

### 5.1 左栏：WorkspacePanel

左栏负责工作区导航，包含三部分：

- `WorkspaceHeader`
  - 显示当前工作区名称
  - 提供完整路径 tooltip
  - 提供“打开工作区”按钮
- `FileTree`
  - 以可展开目录树展示当前工作区内容
  - 高亮当前选中的文件
  - 高亮已打开标签对应的文件
- `SessionHistorySection`
  - 位于左栏底部
  - 默认可折叠
  - 承载现有会话历史、搜索、批量管理能力

### 5.2 中栏：ChatPanel

中栏继续承载聊天能力，但职责收窄：

- 顶部显示当前会话基础信息
- 主体显示消息流
- 底部保留输入、附件和发送能力
- 不再承担工作目录展示和文件预览职责

当没有选中会话时，中栏显示“聊天空状态”，可以复用现有欢迎页的启动会话能力，但布局应适配窄列场景。

### 5.3 右栏：FileWorkbench

右栏作为主工作视图，负责 Markdown 文件处理：

- 空状态：未打开文件时提示用户从左侧选择 Markdown 文件
- 多标签栏：显示已打开的 Markdown 文件
- 工具栏：显示文件名、编辑/预览切换、保存状态、手动保存按钮
- 编辑区：用于修改 Markdown 文本
- 预览区：用于渲染 Markdown 内容

当前常驻的 `ContextPanel` 不再以独立右栏存在，其工作目录展示能力被左栏头部取代，其他能力后续按需迁移或折叠进聊天区。

## 6. 状态模型

本次设计将状态拆成三层，以避免工作区、会话和文件编辑互相污染。

### 6.1 WorkspaceState

全局工作区状态，建议新增独立状态域：

- `currentWorkspacePath: string | null`
- `workspaceTree: WorkspaceTreeNode[]`
- `expandedPaths: Set<string>`
- `selectedPath: string | null`
- `historyCollapsed: boolean`
- `treeLoadingState: 'idle' | 'loading' | 'error'`
- `treeError: string | null`

职责：

- 表示当前应用正在围绕哪个工作区工作
- 维护文件树结构与展开状态
- 驱动左栏展示
- 作为新会话默认 `cwd` 的来源

### 6.2 ChatState

沿用现有会话模型，但明确联动规则：

- 新建会话默认继承 `currentWorkspacePath`
- 已有会话继续保留自己的 `cwd`
- 切换工作区不会自动改写旧会话的 `cwd`
- 历史会话列表仅作为入口，不作为工作区状态源

这样可以避免“切换工作区后历史会话被错误迁移到新目录”的高风险行为。

### 6.3 FileWorkbenchState

右侧工作台状态独立维护：

- `openTabs: FileTab[]`
- `activeTabPath: string | null`
- `viewModeByPath: Record<string, 'edit' | 'preview'>`
- `draftContentByPath: Record<string, string>`
- `savedContentByPath: Record<string, string>`
- `dirtyByPath: Record<string, boolean>`
- `savingByPath: Record<string, boolean>`
- `saveErrorByPath: Record<string, string | null>`
- `lastSavedAtByPath: Record<string, number | null>`

其中 `FileTab` 建议至少包含：

- `path`
- `name`
- `workspacePath`
- `lastOpenedAt`

职责：

- 管理多标签
- 管理草稿与已保存内容
- 管理预览/编辑模式
- 管理自动保存与错误状态

## 7. 工作区持久化与启动门禁

### 7.1 持久化策略

当前仓库已有两个相关概念：

- `config.defaultWorkdir`：可持久化配置
- `currentWorkingDir`：运行时 UI 状态

本次设计将两者统一为“当前工作区”的同一语义：

- 启动时优先从 `config.defaultWorkdir` 读取工作区路径
- 如果路径存在且可访问，则把它作为 `currentWorkingDir`
- 如果路径为空、目录不存在或权限不足，则进入“选择工作区”门禁
- 用户通过“打开工作区”切换目录后，更新 `currentWorkingDir`，并同步持久化到 `config.defaultWorkdir`

### 7.2 启动行为

- 有有效历史工作区：直接进入三栏主界面
- 无有效历史工作区：先显示工作区选择流程，再进入主界面

这里的“门禁”只控制工作区，不阻塞配置、主题等其他已存在的应用初始化逻辑。

## 8. 文件树与实时刷新设计

### 8.1 文件树加载

文件树采用按目录递归展示的 Explorer 结构：

- 根节点为当前工作区
- 文件夹支持展开/折叠
- 首次进入时默认展示根目录一层
- 展开某个目录时读取其子节点

这样可以避免一次性加载大型工作区的全部树结构。

### 8.2 实时刷新

“实时显示这个文件夹下的文件”需要工作区变更监听。仓库中已包含 `chokidar`，因此主进程负责：

- 为当前工作区建立 watcher
- 当文件新增、删除、重命名或目录变化时，向渲染进程推送工作区树刷新事件
- 切换工作区时，销毁旧 watcher，绑定新 watcher

第一期刷新策略采用“目录级重新加载”而不是精细增量树 patch。这样逻辑更稳，足以支撑文件树实时更新。

### 8.3 文件类型规则

- 点击 `.md` 文件：允许在右侧工作台打开
- 点击非 `.md` 文件：不打开标签，只展示“当前仅支持 Markdown 文件”的提示

## 9. Markdown 工作台设计

### 9.1 多标签模型

右侧支持多标签打开多个 Markdown 文件：

- 重复点击已打开文件时，切换到已有标签
- 新文件点击时，新建标签
- 标签可关闭
- 当前活动标签决定工具栏、编辑区和预览区内容

### 9.2 编辑与预览

每个标签在自身维度维护展示模式：

- `edit`
- `preview`

切换模式不创建新标签，也不改变标签顺序。

Markdown 预览复用仓库已有的 `react-markdown` 渲染能力，避免引入第二套渲染栈。

### 9.3 保存策略

每个标签保存策略如下：

- 输入后立即更新 `draftContent`
- 通过防抖自动保存写盘
- 同时保留显式“保存”按钮
- 标签展示未保存标记
- 手动保存和自动保存共用同一套写盘逻辑

### 9.4 外部修改冲突

第一期采用最小安全策略：

- 如果磁盘文件已发生变化且当前标签也有未保存草稿，则保存时中止覆盖
- 向用户提示文件冲突
- 保留当前草稿，不静默丢失

本次不实现 diff 合并能力。

## 10. 关键交互流

### 10.1 启动进入应用

1. 读取持久化工作区路径
2. 路径有效则进入三栏界面并加载文件树
3. 路径无效则要求用户选择工作区
4. 选中工作区后写入持久化配置并进入主界面

### 10.2 打开 Markdown 文件

1. 用户点击左侧 `.md`
2. 主进程读取文件内容
3. 若标签已存在，则切换到该标签
4. 若标签不存在，则创建标签并在右侧显示编辑器

### 10.3 编辑与保存

1. 用户修改文本
2. 对应标签进入 `dirty` 状态
3. 防抖自动保存
4. 保存成功后清除 `dirty`
5. 保存失败则保留草稿并显示错误状态

### 10.4 切换工作区

1. 用户点击“打开工作区”
2. 若右侧存在脏标签，先阻止切换并弹确认
3. 用户选择：
   - 保存并切换
   - 放弃更改并切换
   - 取消
4. 确认切换后关闭旧工作区 watcher，加载新工作区文件树，刷新聊天默认工作区来源

### 10.5 使用历史会话

1. 用户在左下角展开历史会话
2. 选中某会话后，中栏切换为对应聊天内容
3. 该会话保留原有 `cwd`
4. 左侧工作区和右侧文件工作台不被历史会话反向改写

## 11. 组件拆分建议

为了避免把新逻辑继续堆入现有 `Sidebar` 与 `ContextPanel`，建议按职责拆分：

- `AppShell`
- `WorkspacePanel`
- `WorkspaceHeader`
- `FileTree`
- `FileTreeNode`
- `SessionHistorySection`
- `ChatPanel`
- `FileWorkbench`
- `FileTabsBar`
- `WorkbenchToolbar`
- `MarkdownEditorPane`
- `MarkdownPreviewPane`
- `EmptyFileState`

现有组件建议的处理方式：

- `Sidebar`
  - 其会话管理逻辑迁移到 `SessionHistorySection`
  - 不再直接作为主左栏使用
- `ChatView`
  - 保留消息流与输入逻辑，外层封装为 `ChatPanel`
- `WelcomeView`
  - 其“新建会话”能力保留，但空状态布局需适配中栏
- `ContextPanel`
  - 不再作为常驻右栏；其与工作目录相关的展示逻辑停止承担主视图职责

## 12. 主进程与 IPC 设计

为支撑文件树和编辑器，需要补充工作区文件相关 IPC。

建议新增能力：

- `workspace.tree.get`
  - 获取某个目录下的子节点
- `workspace.file.read`
  - 读取 Markdown 文件内容
- `workspace.file.write`
  - 写回 Markdown 文件内容
- `workspace.watch.changed`
  - 主进程主动推送工作区树变更事件

同时调整现有工作区相关逻辑：

- `workdir.get`
  - 返回当前持久化后的工作区路径
- `workdir.select`
  - 选择新工作区后，同时更新运行时状态与持久化配置

文件系统访问坚持放在主进程：

- 渲染进程不直接操作 Node 文件系统
- 所有目录树、读写、watcher 都经由主进程桥接
- 这样与现有 Electron 安全边界保持一致

## 13. 错误处理

### 13.1 工作区相关

- 启动恢复的工作区不存在：要求用户重新选择
- 工作区无权限访问：展示错误并拒绝进入主界面
- 文件树加载失败：左栏显示局部错误态和重试入口

### 13.2 文件读写相关

- Markdown 文件读取失败：不打开空标签，直接提示错误
- 自动保存失败：标签保持 `dirty`，保留草稿和错误标记
- 手动保存失败：保留当前草稿并提示失败原因
- 已打开的 Markdown 文件被外部删除：对应标签进入不可保存的错误态，并提示用户关闭该标签或重新选择文件

### 13.3 切换保护

- 关闭脏标签时弹确认
- 切换工作区且存在脏标签时弹确认
- 第一期开关选项固定为：
  - 保存并继续
  - 放弃更改并继续
  - 取消

## 14. 测试范围

第一期验收重点围绕主链路：

### 14.1 启动与持久化

- 有有效历史工作区时自动恢复
- 无有效工作区时进入选择流程
- 切换工作区后持久化成功

### 14.2 左栏文件面板

- 目录树可加载
- 文件夹可展开/折叠
- 文件变化后树结构可刷新
- 底部历史会话区可折叠

### 14.3 文件工作台

- 点击 `.md` 能打开标签
- 重复点击不会重复开标签
- 多标签切换正常
- 编辑/预览切换正常
- 自动保存与手动保存都可写盘
- 未保存状态展示正确

### 14.4 边界行为

- 点击非 `.md` 只提示不打开
- 关闭脏标签时会拦截确认
- 切换工作区时若有脏标签会拦截确认
- 保存冲突时不会静默覆盖

### 14.5 聊天联动

- 新建会话默认使用当前工作区
- 历史会话仍能正常加载
- 工作区切换不会破坏旧会话的 `cwd`

## 15. 验收标准

满足以下条件即可视为本次设计落地成功：

- 应用启动后可以稳定进入某个工作区
- 左栏成为工作区文件导航，而不是会话历史主栏
- 中栏保留稳定可用的聊天体验
- 右栏可以多标签编辑与预览 Markdown 文件
- 工作区切换、自动保存、未保存拦截三条关键链路稳定

## 16. 推荐实施方向

推荐按“工作区优先的三栏重组方案”实施，而不是在现有 `Sidebar` / `ContextPanel` 上继续叠补丁。

原因如下：

- 更符合用户明确要求的 VS Code 式结构
- 状态边界清晰，后续扩展更稳
- 能复用现有聊天能力与 Markdown 渲染能力
- 为后续支持更多文件类型、聊天引用当前文件、文件级 AI 操作留下清晰扩展点

## 17. 实施前提

进入实现阶段前，默认以下决定已经锁定：

- 文件树是左栏主结构
- 历史会话固定下沉到左栏底部
- 右栏第一期只支持 Markdown
- 保存策略为自动保存优先，同时保留手动保存
- 多标签为第一期正式能力，不延后

以上约束后续若要调整，应先更新本设计文档，再生成新的实施计划。
